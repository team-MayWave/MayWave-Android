# GitHub PR 자동 생성 스킬

## Purpose

GitHub Pull Request를 실무형 컨벤션에 맞춰 자동으로 생성한다.

이 스킬은 현재 브랜치명, 커밋 메시지, 변경 파일, PR 템플릿을 기반으로 PR 제목과 본문을 생성하고 GitHub CLI를 사용해 PR을 생성하는 데 사용한다.

---

## Main Rule

PR 생성 시 다음 규칙을 반드시 지킨다.

- PR 제목은 실무형 컨벤션을 따른다.
- PR 제목은 작업 타입, 작업 범위, 작업 핵심, 이슈 번호를 포함한다.
- PR 본문은 제공된 PR 템플릿을 그대로 사용한다.
- PR 본문은 한글로 작성한다.
- 관련 이슈가 있으면 `Closes #이슈번호`를 작성한다.
- 작업 내용은 bullet point 형식으로 구체적으로 작성한다.
- 테스트 방법은 실제 실행 기준으로 작성한다.
- PR 본문 파일은 UTF-8로 저장한다.
- GitHub CLI 사용 시 PR 본문은 `--body-file`로 전달한다.

---

## PR 제목 형식

### 이슈 번호가 있는 경우

다음 형식을 사용한다.

```text
type(scope): 제목 (#이슈번호)
```

예시:

```text
feat(auth): 로그인 화면 UI 구현 (#12)
```

---

### 이슈 번호가 없는 경우

브랜치명이나 커밋 메시지에서 이슈 번호를 찾을 수 없는 경우 다음 형식을 사용한다.

```text
type(scope): 제목
```

예시:

```text
feat(auth): 로그인 화면 UI 구현
```

---

## PR 제목 구성 요소

PR 제목은 다음 요소로 구성한다.

```text
type(scope): 제목 (#이슈번호)
```

예시:

```text
feat(auth): 로그인 화면 UI 구현 (#12)
```

각 요소의 의미는 다음과 같다.

| 요소 | 예시 | 의미 |
|---|---|---|
| type | feat | 작업 종류 |
| scope | auth | 작업 범위 |
| 제목 | 로그인 화면 UI 구현 | 작업 핵심 내용 |
| 이슈 번호 | #12 | 연결된 GitHub 이슈 번호 |

---

## PR 타입 규칙

`type`은 다음과 같은 작업 종류를 나타낸다.

| 타입 | 의미 | 사용 상황 |
|---|---|---|
| feat | 새로운 기능 추가 | 화면 추가, API 기능 추가 |
| fix | 버그 수정 | 오류 수정, 잘못된 동작 수정 |
| refactor | 코드 구조 개선 | 기능 변화 없이 구조 개선 |
| style | 코드 스타일 수정 | 포맷팅, 네이밍 수정 |
| docs | 문서 수정 | README, 문서 파일 수정 |
| test | 테스트 코드 수정 | 테스트 추가, 테스트 수정 |
| chore | 기타 작업 | 단순 설정, 관리 작업 |
| build | 빌드 설정 변경 | Gradle, 의존성, 빌드 설정 |
| ci | 자동화 설정 변경 | GitHub Actions, 배포 자동화 |
| perf | 성능 개선 | 렌더링 최적화, 속도 개선 |

---

## PR 범위 규칙

`scope`에는 작업한 주요 영역을 작성한다.

| 범위 | 의미 |
|---|---|
| auth | 로그인, 회원가입, 인증 |
| login | 로그인 화면 또는 로그인 로직 |
| signup | 회원가입 화면 또는 회원가입 로직 |
| ui | 공통 UI |
| navigation | 화면 이동 |
| network | 서버 통신 |
| websocket | 웹소켓 |
| firebase | Firebase 설정 |
| gradle | Gradle 설정 |
| theme | 색상, 테마 |
| home | 홈 화면 |
| profile | 프로필 화면 |
| app | 앱 전체 구조 |
| common | 공통 코드 |
| core | 핵심 공통 모듈 |

작업 범위가 명확하지 않으면 다음 중 하나를 사용한다.

```text
app
common
core
```

## PR 제목 작성 규칙

PR 제목은 다음 규칙을 따른다.

- 제목 내용은 한글로 작성한다.
- `type`과 `scope`는 컨벤션 용어이므로 영어 사용을 허용한다.
- 제목은 간결하게 작성한다.
- 작업 핵심만 포함한다.
- 불필요한 설명을 넣지 않는다.
- 파일명, 클래스명, 함수명은 영어 사용을 허용한다.

좋은 예시:

```text
feat(auth): 로그인 화면 UI 구현 (#12)
fix(login): 로그인 버튼 클릭 오류 수정 (#23)
refactor(network): Retrofit 생성 구조 분리 (#31)
build(gradle): Android 서명 설정 추가 (#45)
docs(readme): 실행 방법 문서 추가 (#52)
```

나쁜 예시:

```text
로그인 수정
feat: 이것저것 수정
Add login screen
🔀 :: (#12) - 로그인 화면에서 버튼이랑 텍스트랑 입력창이랑 전체 디자인 수정
```

## PR 본문 규칙

PR 본문은 제공된 PR 템플릿을 그대로 사용한다.

템플릿의 항목 이름은 임의로 삭제하거나 변경하지 않는다.

본문 내용은 한글로 작성한다.

단, 다음 항목은 영어 사용을 허용한다.

- 코드명
- 파일명
- 클래스명
- 함수명
- 브랜치명
- 명령어
- 라이브러리명
- GitHub 예약어

---

## 관련 이슈 작성 규칙

### 이슈 번호가 있는 경우

`관련 이슈` 항목에는 다음 형식을 자동으로 삽입한다.

```markdown
Closes #이슈번호
```

예시:

```markdown
## 관련 이슈

Closes #12
```

`Closes #12`는 PR이 병합될 때 GitHub 이슈 #12를 자동으로 닫기 위해 사용한다.

---

### 이슈 번호가 없는 경우

이슈 번호를 찾을 수 없는 경우 다음과 같이 작성한다.

```markdown
## 관련 이슈

없음
```

---

## 작업 내용 작성 규칙

작업 내용은 bullet point 형식으로 구체적으로 작성한다.

각 항목은 `-` 기호로 시작한다.

예시:

```markdown
## 작업 내용

- 로그인 화면 UI를 구현했습니다.
- 아이디 입력창과 비밀번호 입력창을 추가했습니다.
- 로그인 버튼 컴포넌트를 분리했습니다.
- 화면 이동을 위한 클릭 이벤트 구조를 연결했습니다.
```

작성 규칙:

- 커밋 메시지를 기반으로 실제 작업 내용을 요약한다.
- 변경 파일을 참고해 작업 범위를 구체화한다.
- 너무 추상적으로 작성하지 않는다.
- 리뷰어가 변경 범위를 이해할 수 있도록 작성한다.
- 하나의 bullet point에는 하나의 작업만 작성한다.
- “추가했습니다”, “수정했습니다”, “분리했습니다”, “연결했습니다”, “구현했습니다”처럼 명확한 동사로 끝낸다.

나쁜 예시:

```markdown
## 작업 내용

- 여러 가지 수정
- 코드 정리
```

좋은 예시:

```markdown
## 작업 내용

- 로그인 화면의 전체 레이아웃을 구성했습니다.
- 로그인 버튼 컴포넌트를 별도 Composable 함수로 분리했습니다.
- 입력 필드 간 간격과 정렬을 수정했습니다.
```

## 테스트 방법 작성 규칙

테스트 방법은 실제 실행 기준으로 작성한다.

예시:

```markdown
## 테스트 방법

- Android Studio에서 앱을 실행했습니다.
- 로그인 화면이 정상적으로 표시되는지 확인했습니다.
- 아이디와 비밀번호 입력창에 텍스트가 정상적으로 입력되는지 확인했습니다.
- 로그인 버튼 클릭 시 앱이 중단되지 않는지 확인했습니다.
```

작성 규칙:

- 실제로 실행한 방법을 작성한다.
- 단순히 “확인했습니다”라고 작성하지 않는다.
- 리뷰어가 같은 방식으로 확인할 수 있게 작성한다.
- 실행 환경이 필요한 경우 함께 작성한다.

나쁜 예시:

```markdown
## 테스트 방법

- 확인했습니다.
```

좋은 예시:

```markdown
## 테스트 방법

- Android Studio에서 앱을 실행했습니다.
- 홈 화면에서 로그인 화면으로 이동되는지 확인했습니다.
- 로그인 화면의 버튼과 입력창이 정상적으로 표시되는지 확인했습니다.
```

## 이슈 번호 추출 규칙

브랜치명에서 이슈 번호를 추출한다.

다음과 같은 브랜치명에서 이슈 번호를 추출할 수 있다.

```text
feature/#12-login-screen
feature/12-login-screen
feat/#12-login
fix/#34-login-error
chore/56-gradle-setting
```

위 브랜치명에서는 각각 다음 이슈 번호를 추출한다.

```text
#12
#12
#12
#34
#56
```

이슈 번호를 추출할 수 없는 브랜치명 예시:

```text
feature/login-screen
fix/login-error
develop
main
```

이 경우 PR 제목에서 이슈 번호 부분을 생략하고, 관련 이슈에는 `없음`을 작성한다.

## 커밋 메시지 분석 규칙

커밋 메시지를 기반으로 PR 제목과 작업 내용을 생성한다.

커밋 메시지 예시:

```text
feat(auth): 로그인 화면 UI 구현
feat(auth): 로그인 입력창 추가
refactor(auth): 로그인 버튼 컴포넌트 분리
```

생성할 PR 제목 예시:

```text
feat(auth): 로그인 화면 UI 구현 (#12)
```

생성할 작업 내용 예시:

```markdown
## 작업 내용

- 로그인 화면 UI를 구현했습니다.
- 로그인 입력창을 추가했습니다.
- 로그인 버튼 컴포넌트를 분리했습니다.
```

커밋이 여러 개인 경우 가장 핵심적인 작업을 PR 제목으로 사용한다.

---

## 변경 파일 분석 규칙

커밋 메시지만으로 작업 내용을 판단하기 어려운 경우 변경 파일을 함께 확인한다.

예시:

| 변경 파일 | 추론 가능한 scope |
|---|---|
| `LoginScreen.kt` | login |
| `AuthViewModel.kt` | auth |
| `RetrofitClient.kt` | network |
| `ApiService.kt` | network |
| `NavGraph.kt` | navigation |
| `build.gradle.kts` | gradle |
| `google-services.json` | firebase |
| `Theme.kt` | theme |

변경 파일을 기반으로 `scope`를 결정하되, 커밋 메시지와 충돌하면 실제 변경 목적을 우선한다.

## 인코딩 규칙

PR 본문 파일은 반드시 UTF-8로 저장한다.

한글이 포함되므로 인코딩이 잘못되면 PR 본문이 깨질 수 있다.

PowerShell에서 PR 템플릿 또는 PR 본문 파일을 읽을 때는 `-Encoding utf8`을 명시한다.

예시:

```powershell
Get-Content .github/PULL_REQUEST_TEMPLATE.md -Encoding utf8
```

PR 본문 파일을 저장할 때도 UTF-8을 사용한다.

---

## GitHub CLI 사용 규칙

GitHub CLI를 사용해 PR을 생성할 때 PR 본문은 직접 문자열로 전달하지 않는다.

반드시 `--body-file` 옵션을 사용한다.

예시:

```bash
gh pr create \
  --title "feat(auth): 로그인 화면 UI 구현 (#12)" \
  --body-file pr_body.md
```

이유:

- 한글 깨짐을 방지하기 위해서
- 줄바꿈이 있는 Markdown 본문을 안정적으로 전달하기 위해서
- 긴 PR 본문을 명령어에 직접 넣지 않기 위해서

## 동작 순서

PR 생성은 다음 순서로 진행한다.

1. 현재 브랜치명을 확인한다.
2. 현재 브랜치명에서 이슈 번호를 추출한다.
3. 이슈 번호가 없으면 이슈 번호 없는 PR 제목 형식을 사용한다.
4. 커밋 메시지를 확인한다.
5. 변경 파일 목록을 확인한다.
6. 커밋 메시지를 기반으로 작업 타입을 결정한다.
7. 커밋 메시지와 변경 파일을 기반으로 작업 범위를 결정한다.
8. 커밋 메시지와 변경 파일을 기반으로 작업 내용을 요약한다.
9. PR 제목을 생성한다.
10. PR 템플릿을 읽는다.
11. PR 템플릿을 기반으로 PR 본문을 생성한다.
12. PR 본문 파일을 UTF-8로 저장한다.
13. GitHub CLI의 `gh pr create` 명령어로 PR을 생성한다.

### PR 제목

```text

```

```markdown
## 관련 이슈

## 작업 내용

- 아이디 입력창과 비밀번호 입력창을 추가했습니다.
- 로그인 버튼 컴포넌트를 분리했습니다.
- 버튼 클릭 이벤트를 상위 Composable에서 전달받도록 구성했습니다.

## 테스트 방법

- Android Studio에서 앱을 실행했습니다.
- 로그인 화면이 정상적으로 표시되는지 확인했습니다.
- 아이디와 비밀번호 입력창에 텍스트가 정상적으로 입력되는지 확인했습니다.
- 로그인 버튼 클릭 시 앱이 중단되지 않는지 확인했습니다.
```

## 금지 예시

다음과 같은 PR 제목은 생성하지 않는다.

```text
로그인 수정
Add login screen
feat: 이것저것 수정
🔀 :: (#12) - 로그인 화면이랑 이것저것 많이 수정함
```

다음과 같은 형식을 사용한다.

```text
feat(auth): 로그인 화면 UI 구현 (#12)
```
